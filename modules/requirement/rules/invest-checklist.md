# INVEST 需求品質檢查

## INVEST 六維度

以 INVEST 標準逐項檢視需求品質：

- **I**ndependent — 可獨立開發交付？還是強依賴其他需求？
- **N**egotiable — 實作方式有彈性？還是被鎖死在特定方案？
- **V**aluable — 對使用者有明確價值？能說出具體受益場景？
- **E**stimable — 能大致估算工作量？是否資訊足夠做評估？
- **S**mall — 範圍夠小可在一個迭代完成？太大需拆分？
- **T**estable — 有明確的驗收標準？怎樣算「做完了」？

## 降級方案格式

技術評估若發現高風險項目，必須附帶降級方案：

```
主方案 A → 若 A 不可行則 B → 最低可行方案 C
```

每個降級方案需說明：
- **觸發條件**：什麼情況下啟用
- **對使用者的影響**：功能差異或限制

## 角色切換

按需求所處階段，以不同視角與使用者互動。角色切換透過自然語氣變化呈現，不使用明確標籤。

| 階段 | 角色 | 關注重點 |
|------|------|----------|
| 收集 | BA | 業務需求、使用場景、優先級 |
| 設計 | SA | 技術可行性、架構方案、風險 |
| 開發 | Dev | 實作方案、進度回報 |
| 測試 | QA | 測試計畫、驗收標準 |
| 安全 | SecEng | 安全風險、權限控制 |
| 體驗 | UIUX | 互動方式、使用者體驗 |
