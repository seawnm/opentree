# 需求處理流程

```
1. 記錄 → 2. 分類 → 3. 訪談 → 4. 評估 → 5. 規格 → 6. 確認
```

## 步驟 1：記錄原始需求

- 呼叫 `requirement-tool create`，`--raw-text` 保存使用者原話（不可改寫）
- `--requester` 和 `--requester-id` 從 system prompt 取得
- `--source` 記錄來源頻道

## 步驟 2：初步分類

- 判斷 `--type`：影響多人的功能為 `team`，個人工具為 `personal`
- 判斷 `--priority`：Must（核心功能）/ Should（重要）/ Could（加分）/ Won't（暫不做）

## 步驟 3：訪談補齊（資訊不足時）

- 以 BA（業務分析）視角發起訪談
- **一次只問一個問題**，等使用者回答後再問下一題
- 採用 TEDW 提問法：Tell（說說看） → Explain（為什麼） → Describe（怎麼用） → Walk through（走一遍流程）
- **選項需附帶預設值說明**：提供選項時，若功能有預設行為，主動告知使用者「如果你不選，預設會...」
- 訪談完成後，用 `requirement-tool history` 記錄訪談摘要

### 步驟 3 到 4 檢查點：INVEST 快篩

- 訪談結束後，以 INVEST 標準快速檢視（詳見 invest-checklist.md）
- 重點檢查 **E**（Estimable）和 **T**（Testable）
- 若發現不足，回到步驟 3 補問

## 步驟 4：技術評估

- 以 SA（系統分析）視角評估可行性、技術方案、風險
- 技術可行性檢查清單：API/工具限制、降級方案、外部依賴風險
- 用 `requirement-tool history --role SA` 記錄評估結果
- 回覆使用者評估結論

### 步驟 4 到 5 檢查點：降級方案

- 高風險項目必須附帶降級方案：主方案 A → 若不可行則 B → 最低可行方案 C
- 每個降級方案需說明觸發條件和對使用者的影響

## 步驟 5：規格書（確認可行後）

- 更新需求的「規格摘要」區塊（目標、範圍、驗收標準、限制與假設）
- 用 `requirement-tool update --status confirmed` 更新狀態

### 步驟 5 到 6 檢查點：調整範圍確認

- 需求新增「新的交付格式」或「新的功能模組」時，**必須確認適用範圍**
- 不要假設使用者只想用在「當下討論的場景」——主動確認完整適用範圍

## 步驟 6：確認

- 向使用者呈現**完整的需求規格表**（不是摘要，是逐項列出）
- 規格表涵蓋：目標、範圍、交付物清單、格式與適用場景、驗收標準、限制與假設
- **逐項核對**：特別標注本次新增/變更的項目
- 確認後更新狀態為 `in_progress`
- **禁止跳過此步驟**：即使使用者說「就這樣」，仍需呈現完整規格讓使用者過目一次

## 需求狀態流轉

```
draft → confirmed → in_progress → done
                                 ↘ cancelled
```

| 狀態 | 說明 | 觸發時機 |
|------|------|----------|
| draft | 已記錄，待訪談/評估 | create 時自動設定 |
| confirmed | 規格確認，待開發 | 使用者確認規格後 |
| in_progress | 開發中 | 開始實作時 |
| done | 已完成驗收 | 使用者確認完成 |
| cancelled | 已取消 | 使用者或評估後取消 |
